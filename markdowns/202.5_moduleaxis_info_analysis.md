# ModuleAxisInfoAnalysis 原理分析

## 一、概述

`ModuleAxisInfoAnalysis` 是 Triton 编译器中一个**模块级（跨函数）的轴信息分析**。它在整个模块（包含多个函数、存在函数调用）的范围内，推断每个 SSA Value 在每个维度上的**连续性（Contiguity）**、**可除性（Divisibility）** 和**恒常性（Constancy）** 这三个性质，合称 **AxisInfo**。

这些信息在代码生成阶段直接决定访存指令是否可以合并（coalescing）、对齐优化是否可行、向量化宽度等。

---

## 二、AxisInfo 格（Lattice）——分析的基本单位

### 2.1 三个核心属性

| 属性 | 含义 | 数学意义 |
|------|------|----------|
| **Contiguity[d]** | 第 d 维上最短连续整数片段的长度 | 数组可被划分为 N/C 段，每段 C 个连续整数。C 必须是 2 的幂 |
| **Divisibility[d]** | 第 d 维上每组首个元素的最大 2 的幂因子 | 从 Contiguity 分组后，每组第一个数的最大 2 的幂公因子 |
| **Constancy[d]** | 第 d 维上重复相同值的最短片段长度 | 数组可被划分为 N/C 段，每段 C 个值恒等。C 必须是 2 的幂 |

### 2.2 实际意义与示例

这三个属性描述的是 Triton **线程计算出的元素索引（index）或地址偏移量（offset）的数值模式**，直接决定最终访存能否合并、对齐。

**示例一：连续访问（步长 1，32 线程，float32）**

每个线程的地址偏移 = `thread_index * sizeof(float32)` = `thread_index * 4`。

```
线程 tid:  0,  1,  2,  3,  ...,  31
地址偏移:  0,  4,  8,  12, ..., 124   ← 每步 +4 字节
```

| 属性 | 值 | 含义 |
|------|----|------|
| **Contiguity** | 32 | 32 个线程的偏移量连续递增 |
| **Divisibility** | 4 | 首地址 0 的最大 2 的幂因子是 4（4 字节对齐） |
| **Constancy** | 1 | 没有重复值，每个线程访问不同地址 |

> contiguity=32 → 128 字节连续 → **一次合并为一条大的内存事务（coalescing）**

**示例二：步长为 2 的访问**

```
线程 tid:  0,  1,  2,  3,  ...,  31
地址偏移:  0,  8,  16, 24, ..., 248
```

| 属性 | 值 | 含义 |
|------|----|------|
| **Contiguity** | **1** | 相邻线程地址差 8 字节（> 元素大小 4），**不连续** |
| **Divisibility** | **8** | 首地址 0 的最大 2 的幂因子是 8 |
| **Constancy** | 1 | 没有重复值 |

> contiguity=1 → 无法合并，每条事务只服务一个线程——**性能杀手**

**示例三：所有线程访问同一地址**

```
线程 tid:  0,  1,  2,  3,  ...,  31
地址偏移:  42, 42, 42, 42, ..., 42
```

| 属性 | 值 | 含义 |
|------|----|------|
| **Contiguity** | 1 | 不是连续递增序列 |
| **Divisibility** | 2 | 42 的最大 2 的幂因子是 2 |
| **Constancy** | **32** | 32 个线程的值完全相同 → **一次读取广播给所有线程** |

### 2.3 与硬件访存的对应关系

| AxisInfo 属性 | 映射到硬件的什么？ | 影响的结果 |
|---------------|-------------------|-----------|
| **Contiguity** | 连续线程的地址是否连续递增 | contiguity >= 向量宽度 → coalescing 合并；contiguity=1 → 标量访存 |
| **Divisibility** | 首地址的 2 的幂对齐因子 | divisibility >= 元素字节数 → 自然对齐；>= 缓存行大小 → 更宽的访存指令 |
| **Constancy** | 连续线程是否访问同一地址 | constancy = N → N 个线程共享一次读取（broadcast） |

#### Divisibility 如何决定地址对齐和指令选择

Divisibility 通过 `ptr = base + index * elem_size` 的计算，直接决定地址的对齐字节数：

```
index (tid) = 0, 4, 8, 12, ...         → index divisibility = 4
elem_size   = 4 (float32)
ptr         = base + tid * 4
                                    地址偏移 divisibility = 4 × 4 = 16 字节
                                    也就是说所有地址都是 16 字节对齐的

如果 index divisibility = 1（比如随机索引）:
ptr = base + random * 4
                                    地址偏移 divisibility = 1 × 4 = 4 字节
                                    只能保证 4 字节对齐（自然对齐，但不是宽对齐）
```

对齐值在代码生成阶段直接决定选哪条访存指令：

```cpp
// LowerLoops.cpp:486 — 下游代码生成的实际使用
unsigned vec = axisInfoAnalysis.getContiguity(ptr);
// getContiguity 内部 = min(layout_contig_per_thread, alignment)
// vec 决定了向量化宽度：一次加载多少个元素

// 实际效果：
//   vec = 1  → 用标量 load (ld32)          — 一条指令 1 个元素
//   vec = 2  → 用向量 load (ld64)          — 一条指令 2 个元素
//   vec = 4  → 用向量 load (ld128)         — 一条指令 4 个元素
//   vec = 8  → 用更宽的向量 load (ld256)    — 一条指令 8 个元素
```

对齐要求与可用指令的关系：

| 对齐要求 | 可用的访存指令 | 影响因素 |
|---------|--------------|---------|
| 4 字节对齐 | `ld32` (1 元素) | divisibility >= 1 |
| 8 字节对齐 | `ld64` (2 元素) | divisibility >= 2 |
| 16 字节对齐 | `ld128` (4 元素) | divisibility >= 4 |
| 32 字节对齐 | `ld256` (8 元素) | divisibility >= 8 |

Divisibility 的完整链路：

```
AxisInfoAnalysis
  ↓ 分析出 index 的 divisibility = D
ptr = base + index * elem_size
  ↓ 地址偏移的对齐字节数 = D × elem_size
getAlignment()
  ↓ 实际对齐值（考虑了 contiguity 的上限）
getContiguity()
  ↓ 向量化宽度 vec
代码生成选择 ld32 / ld64 / ld128 / ld256
```

### 2.4 AxisInfo 与 Layout 的关系

**Layout 定义了"逻辑上"的线程→元素映射，AxisInfo 分析了"实际上"的地址偏移数值规律，最终查询把两者结合起来。**

```
┌──────────────────────────────────────────────────────────────────┐
│  Triton 程序                                                      │
│                                                                    │
│  1. Layout (如 BlockedLayout) 定义:                                 │
│     "每个线程拿到 tensor 中的哪些逻辑元素"                              │
│     例: sizePerThread=[2,1], threadsPerWarp=[16,2], warpsPerCTA=[1,4] │
│                                                                    │
│  2. Compiler 根据 Layout 生成地址计算代码:                              │
│     offsets = some_function_of(tid, pid, layout)                    │
│     ptr = base_ptr + offsets * stride * elem_size                   │
│                                                                    │
│  3. AxisInfoAnalysis 分析 offsets/ptr 这个 SSA Value:               │
│     不关心 layout，只追踪数值本身的 contiguity/divisibility/constancy   │
│                                                                    │
│  4. getContiguity/getAlignment 查询方法:                             │
│     同时读取 AxisInfo 结果 和 Layout 的 contigPerThread              │
│     取两者最小值作为最终答案                                          │
└──────────────────────────────────────────────────────────────────┘
```

#### 代码证明：getContiguity 如何同时使用 AxisInfo 和 Layout

```cpp
// AxisInfo.cpp:1253-1270
unsigned ModuleAxisInfoAnalysis::getContiguity(Value offsetsValue,
                                               unsigned elementBitWidth) {
  // 第 1 步：从 Layout 读取"每个线程连续的元素个数"
  auto linAttr = gpu::toLinearEncoding(tensorTy);
  // 将任何 DistributedEncoding (Blocked, MMA, ...) 统一转换为 LinearEncoding
  auto order = linAttr.getOrder();              // 各维度的访存优先级顺序
  auto uniqueContigPerThread = linAttr.getContigPerThread();
  unsigned contiguity = uniqueContigPerThread[order[0]];
  // 取最优先维度的 contiguity

  // 第 2 步：从 AxisInfo 读取"实际数值计算能保证的对齐"
  unsigned align = getAlignment(offsetsValue, elementBitWidth);

  // 第 3 步：取两者的最小值作为保守估计
  contiguity = std::min(align, contiguity);
  // Layout 说"每个线程理想情况下有 N 个连续元素"
  // AxisInfo 说"你算出的偏移量只能保证 M 的对齐"
  return contiguity;
}
```

#### 具体对照

假设 `BlockedLayout`，`sizePerThread=[4, 1]`（每个线程在 dim 0 上连续拿 4 个元素）：

| 场景 | Layout contigPerThread | AxisInfo divisibility | 最终 contiguity（元素） | 访存宽度（float32） |
|------|------------------------|----------------------|-----------------------|-------------------|
| 连续访问 `tid=0,1,2,...` | 4 | ≥4 | **min(4, ≥4) = 4** | 4×4B = **16B → ld128** |
| 步长 2 访问 | 4 | 2 | **min(4, 2) = 2** | 2×4B = **8B → ld64** |
| 随机索引 | 4 | 1 | **min(4, 1) = 1** | 1×4B = **4B → ld32** |

> **关键理解**：Layout 给出的是 upper bound——每个线程至多能连续拿几个元素（逻辑上限）；AxisInfo 给出的是 actual bound——实际索引/偏移量的数值规律（数值下限）。最终取 min。

AxisInfo 本身**完全不知道 Layout 的存在**，它只分析 SSA Value 的数值运算。到查询阶段（`getContiguity` / `getAlignment`），代码生成器才把两者结合。

> **contiguity 与 elem_size 的相互作用**：如果 offset 的 contiguity>1，但乘以非 1 的 elem_size（如 `byte_offset = offset * 4`），则 contiguity 归 1（值序列变为 `[0,4,8,12,...]` 不再是连续整数）。Divisibility 则从 offset 的 divisibility 与 elem_size 的 divisibility 相乘得到。这意味着 contiguity>1 的 offset 乘以非 1 的 elem_size 后，轴信息层面会失去连续性，且 divisibility 不足以补偿，访存合并被破坏。

### 2.5 典型示例

假设二维数组 `shape = [2, 8]`：

```
[[10, 11, 12, 13, 18, 19, 20, 21],
 [20, 21, 22, 23, 28, 29, 30, 31]]
```

- **Contiguity = [1, 4]**：维度 0 上相邻行起始值不连续；维度 1 上每 4 个元素连续（`10,11,12,13` 连续，`18,19,20,21` 连续）
- **Divisibility = [1, 2]**：第一行起始 `10` 的最大 2 的幂因子是 2
- **Constancy = [1, 1]**：没有重复值

### 2.6 格上的 join 操作

```cpp
static AxisInfo AxisInfo::join(const AxisInfo &lhs, const AxisInfo &rhs);
```

对于每个维度 d：
- `contiguity[d] = gcd(lhs.c[d], rhs.c[d])`
- `constancy[d] = gcd(lhs.co[d], rhs.co[d])`
- `divisibility[d]` 使用 `getDivisibilityFromContiguity` 函数——当 contiguity 变化时，除了取 gcd(divisibility)，还要取 gcd(contiguity) 作为更保守的估计

这是标准的格（lattice）join 操作：用 gcd 取两个值的"最大公共信息"，得到同时满足两者的最保守估计。

### 2.7 `highestPowOf2Divisor` 位运算原理

**源码位置**：`triton/include/triton/Dialect/Triton/IR/Utility.h`

```cpp
template <typename T> constexpr T highestPowOf2Divisor(T n) {
  if (n == 0 || n == std::numeric_limits<T>::min())
    return (static_cast<T>(1) << (sizeof(T) * 8 - 2));   // ← n=0 → 返回 2^62
  return (n & (~(n - 1)));                                // ← n≠0 → 提取最低位 1
}
```

##### 核心原理：`n & (~(n - 1))`

这是经典的位运算技巧——**提取最低位的 1（lowest set bit）**。

```
n=12  →   1100
n-1   →   1011
~(n-1) →  0100
n & ~(n-1) = 1100 & 0100 = 0100 = 4    ← 最低位 1 在 bit 2
```

等价于 `1 << countTrailingZeros(n)`：

```
n=12 (1100) → trailing zeros = 2 → 1 << 2 = 4
n=8  (1000) → trailing zeros = 3 → 1 << 3 = 8
n=6  (0110) → trailing zeros = 1 → 1 << 1 = 2
n=1  (0001) → trailing zeros = 0 → 1 << 0 = 1
```

##### 不同 `n` 的运行结果

| n | 二进制 | `highestPowOf2Divisor(n)` | 含义 |
|---|--------|:-------------------------:|------|
| 0 | `0000...0000` | **2^62** | 特殊分支：0 可被任意大 2 的幂整除 |
| 1 | `0000...0001` | **1** | 最低位 1 在 bit0 → 2^0=1 |
| 2 | `0000...0010` | **2** | 只能被 2 整除 |
| 4 | `0000...0100` | **4** | 最低位 1 在 bit2 → 2^2=4 |
| 6 | `0000...0110` | **2** | 6 = 2×3，最大 2 的幂因子是 2 |
| 8 | `0000...1000` | **8** | 能被 8 整除 |
| 12| `0000...1100` | **4** | 12 = 4×3，最大 2 的幂因子是 4 |
| 15| `0000...1111` | **1** | 奇数，只能被 1 整除 |
| 128| `0000...10000000` | **128** | 2^7，128 对齐 |

##### 在 AxisInfo 中的作用

`arange(start, end)` 的 divisibility 直接用 `highestPowOf2Divisor(start)` 初始化：

```cpp
// MakeRangeOp visitor (AxisInfo.cpp:219-226)
getAxisInfo(triton::MakeRangeOp op, ...) {
    auto start = op.getStart();
    auto end = op.getEnd();
    return AxisInfo(/*contiguity=*/{end - start},
                    /*divisibility=*/{highestPowOf2Divisor(start)},
                    /*constancy=*/{1});
}
```

`start` 决定了**第一个元素的地址对齐能力**：start 越大且 2 的幂因子越多，对齐越好，能支持的向量化宽度越大。

##### 实际向量化时的限制链条

`highestPowOf2Divisor` 只是一个初始值，最终向量化宽度在 `getNumElementsPerThread` 中被逐步限制：

```
  初始: divisibility = 2^62  (从 arange(0, 128) 的 start=0 推导)
    ↓
  除以字节宽: 2^62 / 4 = 2^60
    ↓
  min 与 contiguity: min(2^60, 128) = 128
    ↓
  min 与 shapePerCTA: 通常也 ≥ 128
    ↓
  alignment = 128
    ↓
  perThread = min(128, 128 bit / 32 bit) = min(128, 4) = 4
    ↓
  最终向量化宽度: 4 个 f32 = 128 bit
```

所以 `2^62` 实际含义是 **"没有对齐限制"**，向量化瓶颈在 contiguity 和硬件 128bit 向量宽度。

### 2.8 与物理 tensor 排布的关系（以 2D 测试 kernel 为例）

下面用 2D kernel 的真实数据流，逐层追踪三个属性如何映射到物理内存。此例源自 InferCoalescedEncodings pass 的 IR dump，展示了 AxisInfo 如何最终决定 `buildCoalescedEncoding` 的决策。

#### 物理内存排布

测试参数：输入 tensor shape = `[1000, 2000]`，dtype = `f32`，row-major。

```
物理内存（一维线性地址空间）:
                                                                   ... 下一行
  element[0,0]  element[0,1]  element[0,2]  ...  element[0,1999]  element[1,0]
  ──────┬──────────┬──────────┬──────────────────┬──────────────┬───
        │          │          │                  │
     addr+0    addr+4    addr+8           addr+7996   addr+8000
        │          │          │                  │          (跨 2000 个元素)
     连续的 f32  ←┴──────────┴──────────┴───────┘               
     步长=1 个元素 = 4 字节                  ↑ 不连续，步长=2000 元素 = 8000 字节
```

| 维度 | 元素数 | 物理步长（element 单位） | 是否连续 |
|------|--------|--------------------------|----------|
| axis=1（列/y） | 2000 | 1 | **连续** |
| axis=0（行/x） | 1000 | 2000 | **不连续** |

#### Step 1: `arange` 产生连续整数序列

假设 kernel 使用 `XBLOCK=128`、`YBLOCK=256` 的 tile 大小：

```mlir
// AxisInfo: contiguity=[128], divisibility=[2^62]（start=0 → 无对齐限制）, constancy=[1]
%indices_x = tt.make_range {end = 128, start = 0} : tensor<128xi32, ...>

// AxisInfo: contiguity=[256], divisibility=[2^62]（start=0 → 无对齐限制）, constancy=[1]
%indices_y = tt.make_range {end = 256, start = 0} : tensor<256xi32, ...>
```

**Contiguity 含义**：沿着该轴，每多少个元素构成一组连续整数。

| AxisInfo | contiguity | 含义 |
|----------|------------|------|
| `indices_x` | `[128]` | 整个 128 个元素，值从 0 到 127，全部连续 |
| `indices_y` | `[256]` | 整个 256 个元素，值从 0 到 255，全部连续 |

#### Step 2: 乘 stride 后 — 物理地址的雏形

对 row-major tensor，`xstride_in = ynumel = 2000`，`ystride_in = 1`：

```python
in_offsets = xstride_in * indices_x[:, None] + ystride_in * indices_y[None, :]
# in_offsets[i][j] = 2000 * i + 1 * j
```

经过 `expand_dims` → `broadcast` → `arith.muli` → `arith.addi` 后的 contiguity：

```mlir
// indices_x 经 expand_dims{axis=1}: shape [128] → [128, 1]
// contiguity = [128, 1]
// indices_y 经 expand_dims{axis=0}: shape [256] → [1, 256]
// contiguity = [1, 256]

// 最终 in_offsets 形状 [128, 256]
// contiguity = [gcd(128, 1), gcd(1, 256)] = [1, 256]
//
//   in_offsets[i][j] = 2000*i + j
//   沿 axis=0: 相邻行地址差 2000 → 不连续 → contiguity=1
//   沿 axis=1: 相邻列地址差 1     → 连续   → contiguity=256
```

**这就是 AxisInfo contiguity 与物理连续性的直接映射**：

| 维度 | 偏移计算公式 | 地址步长 | `contiguity` |
|------|-------------|----------|:------------:|
| axis=0（行） | `2000 * i` | 2000 个元素 = 8000 字节 | **1** |
| axis=1（列） | `1 * j` | 1 个元素 = 4 字节 | **256** |

#### Step 3: `buildCoalescedEncoding` 的决策

得到 ptr tensor 的 contiguity 后：

```
1. argSort([1, 256]) -> order = [1, 0]
   （contiguity 最高的 axis=1 成为最内层 order[0]）

2. divisibility[order[0]] = axis 1 的 divisibility
    -> 经过 stride 传播后 divisibility 仍为 256

3. alignment = min(divisibility / elemBytes, contiguity, shapePerCTA[order[0]])
             = min(256 / 4, 256, 256)
             = min(64, 256, 256)
             = 64

4. perThread = min(alignment, 128 / elemNumBits)
             = min(64, 128 / 32)
             = min(64, 4)
             = 4
   -> sizePerThread = [1, 4]
```

最终推导出的 `BlockedEncodingAttr`：

```mlir
#blocked = #ttg.blocked<{
  sizePerThread = [1, 4],     <- axis 1 上每线程 4 个连续 f32 = 128bit 向量化
  threadsPerWarp = [1, 32],   <- warp 内 32 个线程沿 axis 1 连续排列
  warpsPerCTA = [2, 2],       <- 4 个 warp 在 2x2 网格覆盖 tile
  order = [1, 0]              <- axis 1（物理连续轴）为最内层
}>
```

#### 三个属性各自的作用总结

| 属性 | 作用对象 | 与物理内存的关系 | 对最终 layout 的影响 |
|------|----------|-----------------|---------------------|
| **Contiguity** | offset 的整数值序列 | 直接反映物理地址连续性：contiguity > 1 的轴一定是物理连续的轴 | 决定 `order`—contiguity 最高的轴成为最内层（`order[0]`） |
| **Divisibility** | 每个连续块的起始地址对齐 | 反映物理地址的对齐程度 | 决定 `sizePerThread`—与 contiguity 一起限定向量化宽度 |
| **Constancy** | 重复值的范围 | 与物理地址无关，反映多线程计算相同值 | 间接影响：在 mask 对齐计算中保证安全向量化宽度 |

核心一句话：**Contiguity 是 AxisInfo 对 "offset 整数序列连续程度" 的描述，通过 `addptr` / `splat` + 乘法传播到 ptr tensor，最终等价于物理地址的连续性**。

## 三、CallGraph 基类——调用图基础设施

`ModuleAxisInfoAnalysis` 继承自 `CallGraph<AxisInfoMapT>`，其中 `AxisInfoMapT = DenseMap<Value, AxisInfo>`。

### 3.1 调用图构造：build 方法

构造函数初始化列表调用基类构造函数，触发 `build()`：

```cpp
// Utility.h: 281
explicit CallGraph(ModuleOp moduleOp) : moduleOp(moduleOp) { build(); }
```

`build()` 完成调用图的全部底层构造，分为两步：

#### 步骤 1：建立调用边（graph）

```cpp
void build() {
    SymbolTableCollection symbolTable;
    DenseSet<FunctionOpInterface> visited;
    moduleOp.walk([&](Operation *op) {
        auto caller = op->getParentOfType<FunctionOpInterface>();
        if (auto callOp = dyn_cast<CallOpInterface>(op)) {
            auto *callee = callOp.resolveCallableInTable(&symbolTable);
            auto funcOp = dyn_cast_or_null<FunctionOpInterface>(callee);
            if (funcOp) {
                graph[caller].emplace_back(callOp, funcOp);
                visited.insert(funcOp);
            }
        }
    });
```

对每个 `CallOpInterface` 操作：
- 通过 `resolveCallableInTable(&symbolTable)` 解析被调用函数符号
- 在 `graph` 中添加一条从 caller 到 callee 的边，记录具体 callOp
- 用 `visited` 记录所有被调用过的函数

> **func 与 symbol 的区别**：`resolveCallableInTable` 做**符号查找**——按名字找到那个叫 `@foo` 的操作；`dyn_cast<FunctionOpInterface>` 做**类型筛选**——只保留函数，忽略其他符号。前者是查找工具，后者是类型约束。

#### 步骤 2：识别根函数（roots）

```cpp
    moduleOp.walk([&](FunctionOpInterface funcOp) {
        if (!visited.count(funcOp))
            roots.push_back(funcOp);
    });
}
```

未被任何 call op 调用的函数标记为**根函数（root）**，是后续 walk 遍历的起始点。

如果模块只有 `func_a` 和 `func_b`，且 `func_a` 调用了 `func_b`，则 `roots = [func_a]`，`graph[func_a] = [(callOp, func_b)]`。

> **为什么不能省略 build？** 即使 Inline 后模块只有 1 个函数、没有 call op，`build()` 仍然必要：它将根函数填充到 `roots` 中作为 walk 起点。若 roots 为空，后续 `walk()` 不会遍历任何函数，整个分析产生零个结果。这是一种 RAII 设计：构造函数完成则对象完全就绪。

### 3.2 核心数据结构

```cpp
ModuleOp moduleOp;  // 所属模块
DenseMap<FunctionOpInterface,
         SmallVector<pair<CallOpInterface, FunctionOpInterface>>> graph;  // 调用图
FuncDataMapT funcMap;  // 每个函数关联的 AxisInfoMapT
SmallVector<FunctionOpInterface> roots;  // 根函数集合
```

### 3.3 调用图遍历：walk 方法

`walk` 是整个分析最核心的控制流抽象，由 `doWalk` 递归实现，**同时控制 Edge 和 Node 回调的触发时序**。

| 术语 | 在调用图中的含义 |
|------|----------------|
| **Node（节点）** | 一个函数（`FunctionOpInterface`） |
| **Edge（边）** | 一条调用关系 `(CallOpInterface, FunctionOpInterface)` |

#### 函数签名

```cpp
template <WalkOrder UpdateEdgeOrder,     // 边回调的触发时序
          WalkOrder UpdateNodeOrder,     // 节点回调的触发时序
          typename UpdateEdgeFn, typename UpdateNodeFn>
void walk(UpdateEdgeFn updateEdgeFn, UpdateNodeFn updateNodeFn);
```

两个模板参数控制回调时序：

| 参数 | PreOrder | PostOrder |
|------|----------|-----------|
| `UpdateEdgeOrder` | 递归进入 callee **之前**触发边回调 | 从 callee 返回**之后**触发 |
| `UpdateNodeOrder` | 递归子节点**之前**触发节点回调 | 子节点全部处理**之后**触发 |

#### doWalk 递归实现

```cpp
void doWalk(funcOp, visited, updateEdgeFn, updateNodeFn) {
    // 检测递归（不支持）
    if (visited.count(funcOp)) report_fatal_error("Cycle detected");

    // ── Pre-Order NODE ──
    if constexpr (UpdateNodeOrder == WalkOrder::PreOrder)
        updateNodeFn(funcOp);

    for (auto [callOp, callee] : graph[funcOp]) {
        // ── Pre-Order EDGE ──
        if constexpr (UpdateEdgeOrder == WalkOrder::PreOrder)
            updateEdgeFn(callOp, callee);

        doWalk(callee, visited, updateEdgeFn, updateNodeFn);

        // ── Post-Order EDGE ──
        if constexpr (UpdateEdgeOrder == WalkOrder::PostOrder)
            updateEdgeFn(callOp, callee);
    }

    // ── Post-Order NODE ──
    if constexpr (UpdateNodeOrder == WalkOrder::PostOrder)
        updateNodeFn(funcOp);

    visited.erase(funcOp);
}
```

> `WalkOrder` 是编译期常量（`constexpr`），`if constexpr` 在编译期消除不执行的分支，无运行时开销。

#### 本分析使用的组合

```cpp
walk<WalkOrder::PreOrder, WalkOrder::PostOrder>(
    [](CallOpInterface callOp, FunctionOpInterface funcOp) {},   // Pre-Order edge: 空操作
    [&](FunctionOpInterface funcOp) {                            // Post-Order node: 收集函数
        funcs.push_back(funcOp);
        funcMap.try_emplace(funcOp, AxisInfoMapT{});
    }
);
```

- **Edge 回调 = PreOrder**：进入 callee 之前执行——这里为空
- **Node 回调 = PostOrder**：所有 callee 处理完后才执行——保证**被调用者先于调用者**加入 funcs 列表

```
遍历顺序（root → a → b, root → c）:
  E(root→a)  [空]
  E(a→b)     [空]
  N(b)       ← 被调用者先
  N(a)       ← 次之
  E(root→c)  [空]
  N(c)
  N(root)    ← 调用者最后
```

#### 四种组合

| EdgeOrder | NodeOrder | 回调顺序 | 典型用途 |
|-----------|-----------|----------|---------|
| PreOrder | PreOrder | N→E→N→… (自顶向下) | 自顶向下收集信息 |
| PreOrder | **PostOrder** | E→E→N→N→… (本例) | **callee 先于 caller** |
| PostOrder | PreOrder | N→E→N→… (边返回时处理) | 特定后处理 |
| PostOrder | PostOrder | E→E→N→… (全部后序) | 后序遍历 |

### 3.4 SymbolTableCollection 的作用

在阶段 2 的循环中：

```cpp
SymbolTableCollection symbolTable;
for (auto funcOp : llvm::reverse(sortedFuncs)) {
    initialize(funcOp, callback);
    funcOp.walk([&](CallOpInterface callOp) {
        auto callee = dyn_cast<FunctionOpInterface>(
            callOp.resolveCallableInTable(&symbolTable));
        update(callOp, callee);
    });
}
```

`SymbolTableCollection` 是一个跨 Region 的**懒加载缓存**：当第一次查询某个 Region 的符号时才构建 `SymbolTable`，后续同一 Region 的查询走缓存。`resolveCallableInTable` 从 callOp 中引用的符号名（如 `@my_helper`），从当前区域向外逐层搜索，找到对应的 `FunctionOpInterface`。

**在 Inline 之后的典型流水线中**（模块通常只有一个函数且无 call op）：
- `funcOp.walk(CallOpInterface)` 找不到任何匹配，update 永不执行
- `SymbolTableCollection` 创建时为空，不被实际查询
- `walk(CallOpInterface)` 是 O(1) 的（MLIR 自动跳过不匹配的操作）

但 `ModuleAxisInfoAnalysis` 的设计本身是一个**不假设 Inline 的通用框架**：
1. 调用者的 AxisInfo 始终先就绪（reverse 后调用者先被分析）
2. `update` 将实参的 AxisInfo 以 gcd 方式写入被调函数参数属性
3. 同一被调用者被多调用者调用时，gcd 保证最保守估计
4. **测试场景**可能直接构造包含多函数和 call 的 module

## 四、ModuleAxisInfoAnalysis 构造函数

构造函数是整个分析的核心，分两大阶段：

```cpp
explicit ModuleAxisInfoAnalysis(ModuleOp moduleOp,
                                axisinfo::CallbackType callback = nullptr)
    : CallGraph<AxisInfoMapT>(moduleOp) {
  // --- 阶段 1: 调用图 Post-Order 遍历，收集函数列表 ---
  SmallVector<FunctionOpInterface> funcs;
  walk<WalkOrder::PreOrder, WalkOrder::PostOrder>(
      [](CallOpInterface, FunctionOpInterface) {},
      [&](FunctionOpInterface funcOp) {
        funcs.push_back(funcOp);
        funcMap.try_emplace(funcOp, AxisInfoMapT{});
      });

  // --- 阶段 2: 逆向拓扑序分析 ---
  SetVector<FunctionOpInterface> sortedFuncs(funcs.begin(), funcs.end());
  SymbolTableCollection symbolTable;
  for (auto funcOp : llvm::reverse(sortedFuncs)) {
    initialize(funcOp, callback);      // 函数内数据流分析
    funcOp.walk([&](CallOpInterface callOp) {
      auto callee = dyn_cast<FunctionOpInterface>(
          callOp.resolveCallableInTable(&symbolTable));
      update(callOp, callee);          // 调用者 → 被调用者传播
    });
  }
}
```

### 4.1 阶段 1：Post-Order 收集函数列表

通过 `walk<PreOrder, PostOrder>` 遍历调用图（详见 [3.3 节](#33-调用图遍历walk方法)）：
- **Pre-Order edge callback**: 空操作
- **Post-Order node callback**: 将每个函数加入 `funcs` 向量，在 `funcMap` 中为其初始化空的 `AxisInfoMapT`

被调用者先于调用者被加入。例如调用图 `root → a → b`，加入顺序为 `[b, a, root]`。

### 4.2 阶段 2：正向拓扑序分析

`SetVector(funcs)` 在保持顺序的基础上去重，`reverse` 将其反转为 `[root, a, b]`——从调用者到被调用者的**正向拓扑序**，保证调用者的信息先就绪。

每个函数先执行 `initialize`（函数内 SSA 数据流分析），然后 walk 所有 call op 执行 `update`（将实参 AxisInfo 传播到被调函数参数属性）。

### 4.3 两种场景的实际行为

**场景 A：多函数调用图（通用设计）**

假设调用图 `root → a → b`，reverse 后 `[root, a, b]`：

```
第 1 轮: funcOp = root
  ① initialize(root)      → 算出 root 内所有 value 的 AxisInfo
  ② walk call ops          → 找到 call a，update(callOp, a)
       → 将 root 实参的 AxisInfo 以 gcd 方式写入 a 的参数属性

第 2 轮: funcOp = a
  ① initialize(a)          → getPessimisticValueState 读取 a 的参数属性
                              ↑ 这里读到的是第 1 轮 update 写进去的值
  ② walk call ops          → 找到 call b，update(callOp, b)

第 3 轮: funcOp = b
  ① initialize(b)          → 读到 a 传播过来的参数属性
  ② walk call ops          → 没有 call op，什么都不做
```

**场景 B：Inline 之后（当前的实际路径）**

`funcs = [唯一的函数]`，reverse 后仍是 `[唯一的函数]`：

```
第 1 轮（也是唯一一轮）: funcOp = 唯一的函数
  ① initialize(唯一的函数)      → 函数内数据流分析
  ② walk call ops              → 没有 call op，什么都不做
```

**`update` 的跨函数传播机制在当前路径上从不触发。** 整个分析退化为单函数内的 AxisInfo 数据流分析。

## 五、initialize 方法——函数内轴信息分析

```cpp
void ModuleAxisInfoAnalysis::initialize(FunctionOpInterface funcOp,
                                        axisinfo::CallbackType callback) {
  // 步骤 1：创建数据流求解器并注册 AxisInfoAnalysis
  std::unique_ptr<DataFlowSolver> solver = createDataFlowSolver();
  AxisInfoAnalysis *analysis = solver->load<AxisInfoAnalysis>(callback);

  // 步骤 2：对嵌套隔离区域独立求解
  WalkResult result =
      funcOp.walk<mlir::WalkOrder::PreOrder>([&](Operation *op) {
        if (op->hasTrait<OpTrait::IsIsolatedFromAbove>() &&
            failed(solver->initializeAndRun(op)))
          return WalkResult::interrupt();
        return WalkResult::advance();
      });
  if (result.wasInterrupted()) return;

  // 步骤 3：从 solver 中提取结果存入 funcMap
  auto *axisInfoMap = getFuncData(funcOp);
  auto updateAxisInfoMap = [&](Value value) {
    auto axisInfo = analysis->getLatticeElement(value)->getValue();
    AxisInfo curAxisInfo;
    if (axisInfoMap->count(value))
      curAxisInfo = AxisInfo::join(axisInfo, axisInfoMap->lookup(value));
    else
      curAxisInfo = axisInfo;
    (*axisInfoMap)[value] = curAxisInfo;
  };
  funcOp.walk([&](Operation *op) {
    for (auto value : op->getResults()) updateAxisInfoMap(value);
  });
  funcOp.walk([&](Block *block) {
    for (auto value : block->getArguments()) updateAxisInfoMap(value);
  });
}
```

### 5.1 创建 DataFlowSolver 与加载分析

`createDataFlowSolver()` 加载了两个通用基础分析：

```cpp
std::unique_ptr<DataFlowSolver> createDataFlowSolver() {
  auto solver = std::make_unique<DataFlowSolver>();
  solver->load<dataflow::DeadCodeAnalysis>();           // 第 1 个
  solver->load<dataflow::SparseConstantPropagation>();  // 第 2 个
  return solver;
}
```

然后 `initialize()` 加载第三个分析 `AxisInfoAnalysis`（第 3 个）。

三个分析的分工：

| 分析 | 类型 | Lattice | 职责 |
|------|------|---------|------|
| `DeadCodeAnalysis` | Dense 前向 | `Lattice<Executable>`（每个 Block/Edge） | 标记代码块存活状态 |
| `SparseConstantPropagation` | Sparse 前向 | `Lattice<ConstantValue>`（每个 SSA Value） | 常量传播（DCA 依赖它判断分支条件） |
| `AxisInfoAnalysis` | Sparse 前向 | `Lattice<AxisInfo>`（每个 SSA Value） | **核心**：计算 contiguity/divisibility/constancy |

**依赖链**：AxisInfoAnalysis → DeadCodeAnalysis → SparseConstantPropagation

详细联动机制见[第九章](#九dataflowsolver-联动详解)。

### 5.2 对嵌套隔离区域独立求解

```cpp
funcOp.walk<PreOrder>([&](Operation *op) {
    if (op->hasTrait<OpTrait::IsIsolatedFromAbove>() &&
        failed(solver->initializeAndRun(op)))
        return WalkResult::interrupt();
    return WalkResult::advance();
});
```

对每个 **`IsIsolatedFromAbove`** 的操作内部区域独立运行一次 `initializeAndRun`。典型场景：

| 匹配的操作 | 说明 |
|-----------|------|
| **`tt.func`**（外层 funcOp 自身） | Pre-Order walk 最先访问 funcOp，对整个函数体运行一次分析 |
| **`ttg.warp_specialize.partitions`**（若存在） | GPU warp specialize 分区容器，需单独求解 |

> `scf.for`、`scf.if` 等控制流操作不携带 `IsIsolatedFromAbove` 特质，其区域不会触发独立的 `initializeAndRun`，而是通过外层 funcOp 的分析自动覆盖。

### 5.3 收集所有 Value 的 AxisInfo

Solver 运行完毕后，通过 `analysis->getLatticeElement(value)->getValue()` 提取每个 Value 的最终 AxisInfo，存入 `funcMap[funcOp]` 对应的 `AxisInfoMapT`。如果同一个 value 被多次遇到（如循环迭代），使用 `join`（gcd）合并。

### 5.4 AxisInfoAnalysis 的 Visitor 机制

```cpp
class AxisInfoVisitorList {
    AxisInfo apply(Operation *op, ...) {
        for (auto &visitor : visitors)
            if (visitor->match(op))
                return visitor->getAxisInfo(op, operands);
        return AxisInfo();  // 默认空信息
    }
};
```

`AxisInfoAnalysis` 使用 **Visitor 模式** 根据操作类型分发到不同的 AxisInfo 计算逻辑。注册的 visitor 列表在构造函数中批量注册：

| Visitor | 适用操作 | 计算逻辑摘要 |
|---------|----------|-------------|
| `CastOpAxisInfoVisitor` | ExtSI/ExtUI/TruncI/ConvertLayout/Bitcast | 透传操作数信息 |
| `MakeRangeOpAxisInfoVisitor` | `make_range` | 基于 start/end 直接构造 |
| `ConstantOpAxisInfoVisitor` | `arith.constant` | contiguity=1, constancy=全部, divisibility=值的 2 的幂因子 |
| `AddSubOpAxisInfoVisitor` | addi/subi/addptr | contiguity 取 gcd(constancy 与 contiguity 的交叉组合); divisibility 取 gcd |
| `MulIOpAxisInfoVisitor` | muli | ×1 保留 contiguity; divisibility 相乘 |
| `DivOpAxisInfoVisitor` | divsi/divui | ÷1 保留 contiguity; 特殊 case 保 divisibility |
| `BroadcastOpAxisInfoVisitor` | broadcast | 广播维度设定 constancy=shape |
| `SelectOpAxisInfoVisitor` | select | 条件已知则取对应分支，否则 gcd 合并 |
| `CmpOpAxisInfoVisitor` | cmpi | 基于比较谓词推断结果的 constancy |
| `LoadOpAxisInfoVisitor` | load | 保留 ptr 和 mask 的 constancy, contiguity/divisibility 置 1 |
| `TransOpAxisInfoVisitor` | trans | 应用 permutation 重排各维属性 |

### 5.5 visitOperation 中的 hint 覆盖

```cpp
LogicalResult AxisInfoAnalysis::visitOperation(...) {
    AxisInfo curr = visitors.apply(op, operands);
    // 用 discardable attribute 覆盖计算结果
    AxisInfo::initDimVectorFromHint(
        op->getDiscardableAttr("tt.contiguity"), &newContiguity);
    AxisInfo::initDimVectorFromHint(
        op->getDiscardableAttr("tt.divisibility"), &newDivisibility);
    AxisInfo::initDimVectorFromHint(
        op->getDiscardableAttr("tt.constancy"), &newConstancy);
    curr = AxisInfo(newContiguity, newDivisibility, newConstancy,
                    curr.getConstantValue());
    for (auto *result : results)
        propagateIfChanged(result, result->join(curr));
}
```

允许用户通过 `tt.contiguity`、`tt.divisibility`、`tt.constancy` 这些 discardable attribute 手动覆盖分析结果，为后续 pass 提供更精确的信息。

## 六、update 方法——跨函数边界传播

```cpp
void ModuleAxisInfoAnalysis::update(CallOpInterface callOp,
                                    FunctionOpInterface callee) {
    auto caller = callOp->getParentOfType<FunctionOpInterface>();
    auto *axisInfoMap = getFuncData(caller);
    for (auto entry : llvm::enumerate(callOp->getOperands())) {
        auto value = entry.value();
        auto axisInfo = axisInfoMap->lookup(value);
        if (axisInfo.getRank() != 1) continue;  // 只处理标量参数
        setAttrFn("tt.contiguity", axisInfo.getContiguity(0));
        setAttrFn("tt.divisibility", axisInfo.getDivisibility(0));
        setAttrFn("tt.constancy", axisInfo.getConstancy(0));
    }
}
```

核心逻辑：
1. 找到调用者函数，获取实参的 AxisInfo
2. **只处理标量参数**（rank == 1）：多维 tensor 的轴信息结构复杂，不适合直接传播为函数参数属性
3. 使用 **gcd 合并**：callee 可能被多个 callsite 调用，在参数属性上取 gcd(prevValue, curValue)，保证最保守的公共估计
4. 更新到 callee 的参数属性 `tt.contiguity`/`tt.divisibility`/`tt.constancy`

```
Caller 1 (contiguity=4) --------\
                                  ---> Callee 参数 tt.contiguity = gcd(4, 2) = 2
Caller 2 (contiguity=2) --------/
```

当后续 `initialize(callee)` 运行时，callee 通过 `getPessimisticValueState` 读取到这些参数属性，获得 caller 传播过来的信息。

## 七、查询接口

### 7.1 getAxisInfo

```cpp
AxisInfo *ModuleAxisInfoAnalysis::getAxisInfo(Value value);
```

在 `funcMap` 中查找 value 所属函数的 AxisInfoMap，返回该 value 的 AxisInfo。若不存在返回 nullptr。

### 7.2 getContiguity / getAlignment

```cpp
unsigned getContiguity(Value value);
unsigned getAlignment(Value value);
```

使用 GPU 线性编码（toLinearEncoding）中的 order 确定主维度（order[0]），然后：
- **Contiguity**: `min(alignment, uniqueContigPerThread[order[0]])`
- **Alignment**: `min(maxMultiple, maxContig)`，其中 maxMultiple = divisibility[order[0]] / elemNumBytes（对 ptr tensor）或 divisibility[order[0]]（对普通 tensor）
- **Mask Alignment**: `max(constancy[maskOrder[0]], 1)`

### 7.3 带 elementBitWidth 的重载

```cpp
unsigned getContiguity(Value offsetsValue, unsigned elementBitWidth);
unsigned getAlignment(Value offsetsValue, unsigned elementBitWidth);
```

在 base pointer 和 offset 分离的场景中（如 buffer 操作），需分别计算。此时用 offset 的 contiguity，但用 pointee element type 的 bit width 来计算 alignment，以正确反映实际访存的对齐要求。

### 7.4 运行时连续性计算（getNumElementsPerThread）

在 `buildCoalescedEncoding` 等下游接口中实际使用的不仅是基础的 `getContiguity`，还包括完整的向量化宽度计算链路：

```
getNumElementsPerThread(op, order, axisInfoAnalysis, shapePerCTA):
  1. divisibility = axisInfo.getDivisibility()[order[0]]
  2. contiguity  = axisInfo.getContiguity()[order[0]]
  3. alignment   = min(divisibility / elemBytes, contiguity, shapePerCTA[order[0]])
  4. perThread   = min(alignment, 128 / elemNumBits)   <- 上限 128 位
```

其中 `getContiguity(ptrValue)` 的完整计算：

```
getContiguity(ptrValue):
  1. 从 ptr 的类型获取元素位宽（如 f32 -> 32bit）
  2. 获取该 ptr 在 layout 下的 uniqueContigPerThread
  3. 结合 AxisInfo 中的 divisibility 计算 alignment
  4. contiguity = min(uniqueContigPerThread, alignment, shapePerCTA)
```

这段逻辑将 AxisInfo 的抽象属性（divisibility、contiguity）与硬件约束（128 位向量宽度、layout per-thread 连续性）串联起来，最终得到每线程实际可安全向量化的元素个数。

## 八、整体设计分析

### 8.1 设计要点

1. **调用图 + 正向拓扑序**：`walk<PreOrder, PostOrder>` 以 Post-Order 节点回调得到 `[被调用者…调用者]`，reverse 反转为正向拓扑序，保证调用者信息先就绪并传播给被调用者。
2. **两阶段分析**：函数内用 MLIR 的 `SparseForwardDataFlowAnalysis` 做 SSA 数据流分析；跨函数通过 `update` 将实参信息以参数属性传播。
3. **gcd 作为 join 操作**：所有合并使用最大公约数，保证最保守的正确性。
4. **Visitor + Callback 扩展**：通过 `axisinfo::CallbackType` 允许用户注册自定义 visitor。
5. **Hint override**：discardable attribute 手动覆盖分析结果。
6. **Sparse 分析**：每个 SSA Value 绑定一个 `Lattice<AxisInfo>`，只追踪 Value 级别信息。
7. **Forward 分析**：从 operand → result 单向传播。
8. **Worklist 收敛**：初始化填充第一批 worklist，`propagateIfChanged` 触发 use-def 链传播，直到 worklist 为空（fixpoint）。
9. **保守性**：`join` = gcd，`setToEntryState` = (1,1,1)——不知道时取最保守值。

## 九、DataFlowSolver 联动详解

AxisInfoAnalysis 不是独立运行的。它依赖 `DataFlowSolver` 框架，与 DeadCodeAnalysis (DCA) 和 SparseConstantPropagation (SCP) 协同工作。理解它们的联动关系是理解 AxisInfo 整个数据流求解过程的关键。

### 9.1 类继承层次

```
DataFlowAnalysis                              (框架基类)
  └─ AbstractSparseForwardDataFlowAnalysis     (稀疏前向分析：IR 遍历 + use-def 传播)
       └─ SparseForwardDataFlowAnalysis        (模板层：类型擦除包装)
            └─ AxisInfoAnalysis                (Triton 专用：实现 visitOperation)
```

| 层次 | 关键职责 |
|------|---------|
| `DataFlowAnalysis` | 提供 getOrCreate、addDependency、propagateIfChanged 等核心 API |
| `AbstractSparseForwardDataFlowAnalysis` | 实现 initialize、visit、initializeRecursively、visitBlock 等 IR 遍历和控制流处理逻辑 |
| `SparseForwardDataFlowAnalysis<Lattice<AxisInfo>>` | 模板类型擦除，将 `AbstractSparseLattice` 转为 `AxisInfo *` |
| `AxisInfoAnalysis` | 用户自定义：setToEntryState、visitOperation、visitNonControlFlowArguments |

### 9.2 三个分析的加载与依赖

> **澄清**：DCA 和 SCP **不是 DataFlowSolver 框架内置的分析**。Solver 是一个通用框架，不内嵌任何具体分析。这三个分析都是 Triton 的 `createDataFlowSolver()` / `initialize()` 主动加载的。其他下游分析若需要死代码信息，也应当自行加载 DCA + SCP，而非假设它们已存在。
>
> 同时，**并非所有分析都直接订阅 SCP**。从下文的订阅关系表可以看出：
> - **只有 DCA** 直接订阅了 SCP 的 `Lattice<ConstantValue>`（需要常量值判断分支条件）
> - **AxisInfoAnalysis** 订阅的是 DCA 的 `Lattice<Executable>`（跳过死代码），而非直接订阅 SCP

这里需要区分两个不同的概念：**加载顺序**（初始化顺序）和 **运行时依赖**（重新调度关系），它们由不同的机制控制。

#### 加载顺序：控制初始化先后

```cpp
solver->load<DeadCodeAnalysis>();              // ① 先加载
solver->load<SparseConstantPropagation>();     // ② 次之
solver->load<AxisInfoAnalysis>(callback);      // ③ 最后加载
```

`DataFlowSolver::initializeAndRun` 严格按照加载顺序依次调用每个分析的 `initialize()`（见 `DataFlowFramework.cpp:125`）：

```cpp
for (DataFlowAnalysis &analysis : childAnalyses)
  if (failed(analysis.initialize(top))) return failure();
```

加载顺序重要的原因是：**先初始化的分析会先创建其 Lattice 锚点**。例如 DCA 先初始化，创建了所有 Block 的 `Executable` Lattice，后续 SCP 和 AxisInfoAnalysis 在 `initializeRecursively` 中才能通过 `getOrCreate<Executable>(...)` 获取到它们。

#### 运行时依赖：控制谁的更新触发谁重新调度

依赖关系是在 `initializeAndRun` 的**初始化阶段**，通过各分析在 IR 遍历时动态注册的订阅机制建立的：

| 依赖关系 | 注册点（源码位置） | 机制 | 效果 |
|---------|------------------|------|------|
| DCA → SCP | `DeadCodeAnalysis.cpp:453` `cv->useDefSubscribe(this)` | **useDefSubscribe** | DCA 读取 operand 的 `Lattice<ConstantValue>` 时订阅。当 SCP 更新了该 Lattice，DCA 会被重新调度 |
| SCP → DCA | `SparseAnalysis.cpp:85-86`（基类 `initializeRecursively`） | **blockContentSubscribe** | SCP 和 AxisInfoAnalysis 共享同一个基类，都在 `initializeRecursively` 中通过 `blockContentSubscribe(this)` 订阅每个 Block 的 Executable 状态。当 DCA 将 Block 从 dead 变为 live，SCP 和 AxisInfoAnalysis 都会被重新调度 |
| AxisInfoAnalysis → DCA | `SparseAnalysis.cpp:85-86`（同上，与 SCP 共享） | **blockContentSubscribe** | 同上 |
| AxisInfoAnalysis → SCP | **不直接订阅** | — | AxisInfoAnalysis 有自己的常量体系（`AxisInfo::constantValue`），与 SCP 的 `Lattice<ConstantValue>` 独立。但 SCP 必须加载，因为 DCA 需要它 |

`blockContentSubscribe` 的代码在 `AbstractSparseForwardDataFlowAnalysis::initializeRecursively` 中，所有 Sparse 前向分析（SCP 和 AxisInfoAnalysis）共享同一个基类方法：

```cpp:85:86:mlir/lib/Analysis/DataFlow/SparseAnalysis.cpp
LogicalResult
AbstractSparseForwardDataFlowAnalysis::initializeRecursively(Operation *op) {
  // ...

  for (Region &region : op->getRegions()) {
    for (Block &block : region) {
      // ⬇ 关键：为每个 Block 的 Executable Lattice 注册订阅
      getOrCreate<Executable>(getProgramPointBefore(&block))
          ->blockContentSubscribe(this);
      // ↑ 这里的 this 是指当前的 Sparse 分析实例。
      //   SCP 和 AxisInfoAnalysis 通过同一个基类方法订阅 Executable
      //   当 DCA 把该 Block 从 dead 改为 live 时，所有订阅者都会被重新调度

      visitBlock(&block);
      for (Operation &op : block)
        if (failed(initializeRecursively(&op)))
          return failure();
    }
  }
  // ...
}
```

整个过程是：
1. DCA 先初始化，标记 entry block 为 live，创建了所有 Block 的 `Executable` Lattice
2. 当 SCP 和 AxisInfoAnalysis 初始化时，它们的基类 `initializeRecursively` 遍历 IR 树，对每个 Block 调用 `getOrCreate<Executable>(...)` **获取** DCA 创建的 `Executable` Lattice，同时调用 `blockContentSubscribe(this)` **注册订阅**
3. 之后在 worklist 循环中，如果 DCA 将某个 Block 从 dead 改为 live，该 Block 上所有订阅者（SCP 和 AxisInfoAnalysis）都会被重新调度

同理，`useDefSubscribe` 在每次 `visitOperation` 读取 operand 时注册（`SparseAnalysis.cpp:143`）：

```cpp:141:144:mlir/lib/Analysis/DataFlow/SparseAnalysis.cpp

    AbstractSparseLattice *operandLattice = getLatticeElement(operand);
    operandLattice->useDefSubscribe(this);  // 订阅 operand 的变化
    operandLattices.push_back(operandLattice);
  }
```

#### 总结两套顺序

```
初始化顺序（由加载顺序决定）：  DCA → SCP → AxisInfoAnalysis
运行时订阅关系（谁触发谁）：     AxisInfoAnalysis ←→ DCA ←→ SCP
                                （blockContentSubscribe）  （useDefSubscribe）
```

即：
- **DCA ↔ SCP**：双向依赖。SCP（通过基类 `initializeRecursively` 中的 `blockContentSubscribe`）订阅 DCA 的 `Executable` 状态；DCA（通过 `useDefSubscribe`）订阅 SCP 的 `ConstantValue` 状态
- **AxisInfoAnalysis → DCA**：AxisInfoAnalysis 通过同样的 `blockContentSubscribe` 订阅 DCA 的 `Executable` 状态

#### 双向依赖如何解决顺序问题：Worklist Fixpoint 迭代

DCA 和 SCP 相互订阅看似有循环依赖，但 `DataFlowSolver` 并不需要"先执行哪个"的固定顺序。它使用 **worklist fixpoint 迭代**来解决：

**第一阶段：初始化（有顺序，但只是初始估计）**

DCA 先初始化→标记 entry block 为 live；SCP 再初始化→对当前已知 live 的 block 做常量传播。此时 DCA 尚未完全展开分支，SCP 的常量信息也不完整——都是**保守的初始估计**。

**第二阶段：Worklist 循环（无顺序，反复迭代直到收敛）**

以 DCA 和 SCP 相互触发为例：

```
Step 1: SCP 计算出某条分支条件的常量值（如 %cond = true）
          → 通过 propagateIfChanged 通知 DCA（useDefSubscribe 订阅）

Step 2: DCA 收到通知，重新评估分支
          → 发现 %cond 为 true，标记 then-block 为 live
          → 通过 blockContentSubscribe 通知 SCP 和 AxisInfoAnalysis

Step 3: SCP 收到通知，对 then-block 中的操作做常量传播
          → 可能计算出新的常量值
          → 再通知 DCA ← 回到 Step 1，但这次是新的常量

...重复直到 worklist 为空（fixpoint 收敛）...
```

```
DCA                SCP               AxisInfoAnalysis
 │                  │                     │
 ├─ init ─────────► │                     │   (加载顺序：初始化)
 │                  │                     │
 │◄── useDefSubscribe ──┤                │   (SCP 更新后触发 DCA)
 ├── blockContentSubscribe ──►           │   (DCA 更新后触发 AxisInfoAnalysis)
 └── blockContentSubscribe ──► SCP       │   (DCA 更新后也触发 SCP)
```

**为什么不会死循环？** 因为 lattice 是**单调递增**的——值只会通过 `join`（gcd）向上合并，不会回退。每次重新调度要么推进状态，要么无变化。无变化时 `propagateIfChanged` 不触发，worklist 不会新增条目。由于 lattice 值域有限（contiguity/divisibility/constancy 都是 2 的幂，有上界），最终必然收敛。

> 这正是数据流分析的标准做法：不依赖"顺序正确"，而依赖"迭代收敛"。顺序靠保守初始值起步，精度靠 worklist 推进，终止靠单调性保证。

### 9.3 initializeAndRun 完整流程

```cpp
// DataFlowFramework.cpp:108-145
LogicalResult DataFlowSolver::initializeAndRun(Operation *top) {
  // 阶段 0: 初始化等价 Lattice Anchor
  for (DataFlowAnalysis &analysis : childAnalyses)
    analysis.initializeEquivalentLatticeAnchor(top);  // 默认空操作

  // 阶段 A: 按加载顺序初始化三个分析
  for (DataFlowAnalysis &analysis : childAnalyses)
    if (failed(analysis.initialize(top)))
      return failure();

  // 阶段 B: Worklist 循环直到收敛（fixpoint）
  while (!worklist.empty()) {
    auto [point, analysis] = worklist.front();
    worklist.pop();
    if (failed(analysis->visit(point)))
      return failure();
  }
  return success();
}
```

#### 阶段 A 详解

**① DeadCodeAnalysis::initialize(top)**
```cpp
  analysisScope = top;
  markEntryBlocksLive(top);           // 标记所有 entry block live
  initializeSymbolCallables(top);     // 查找可调用对象
  return initializeRecursively(top);  // 递归遍历 IR
```
- 所有 entry block 标记为 `Executable::live`
- 识别 call/region-branch 操作，创建 `PredecessorState` 记录前驱关系
- **不标记分支后继为 live**（需在 worklist 循环中根据常量条件值完成）

**② SparseConstantPropagation::initialize(top)**

继承自 `SparseForwardDataFlowAnalysis::initialize`：
```cpp
// SparseAnalysis.cpp:55-66
  for (Region &region : top->getRegions()) {
    if (region.empty()) continue;
    for (Value argument : region.front().getArguments())
      setToEntryState(getLatticeElement(argument));
  }
  return initializeRecursively(top);
```
`initializeRecursively` 对每个操作调用 `visitOperation`：
- 检查 DCA 的 `Executable` 状态，跳过 dead block
- 为结果创建 `Lattice<ConstantValue>`，为 operand 注册 useDefSubscribe
- 调用 `visitOperationImpl` 尝试用 `OpFoldResult` 折叠当前操作
- 变化时 `propagateIfChanged` 将使用该结果的操作入 worklist

**③ AxisInfoAnalysis::initialize(top)**

同样继承 `SparseForwardDataFlowAnalysis::initialize`，执行相同的初始化流程，但：
- `setToEntryState` 初始化的是 `Lattice<AxisInfo>`（而非 `Lattice<ConstantValue>`）
- 对函数参数：读取 `tt.contiguity` 等 discardable attribute
- `visitOperationImpl` 调用 `AxisInfoAnalysis::visitOperation`，最终分发到各 Visitor

#### 阶段 B：Worklist 循环

初始化结束后，所有 Lattice 已有第一轮计算结果，worklist 中已有待传播项。

```
while (!worklist.empty()) {
  (point, analysis) = worklist.pop();
  analysis->visit(point)
    └─ for SparseForward: visitOperation(op)
         ├─ 检查 Lattice<Executable> → 死代码则跳过
         ├─ 获取 operands 的 Lattice + useDefSubscribe
         ├─ visitOperationImpl → 计算新值
         └─ propagateIfChanged → 变化则依赖者入 worklist
}
```

#### 初始化后状态示例

```
IR 结构                                        初始化完成后各分析的状态
┌─────────────────────────┐
│ tt.func @kernel() {     │  DCA:    entry block  → Executable=Live
│                         │  SCP:    func 参数     → Lattice<ConstantValue>=unknown
│   %c = arith.constant 4 │         %c              → Lattice<ConstantValue>=4
│   %tid = tt.make_range  │         %tid            → Lattice<ConstantValue>=unknown
│   %i = arith.muli %tid, │         %i              → Lattice<ConstantValue>=unknown
│          %c              │
│                         │  AxisInfo: func 参数     → Lattice<AxisInfo>=pessimistic
│   tt.return             │         %c              → Lattice<AxisInfo>={c=1,d=4,co=1,const=4}
│ }                       │         %tid            → Lattice<AxisInfo>={c=32,d=1,co=1}
└─────────────────────────┘         %i              → Lattice<AxisInfo>={c=1,d=4,co=1}
```

### 9.4 三种触发重新入 worklist 的机制

| 触发机制 | 注册位置 | 触发条件 | 效果 |
|---------|---------|---------|------|
| **① useDefSubscribe** | `visitOperation` 读取 operand 时 | operand 的 Lattice 更新 | 使用该 operand 的操作被重新调度 |
| **② addDependency** | `getLatticeElementFor` 调用时 | 被查询的 Lattice 更新 | 查询者被重新调度 |
| **③ Executable.blockContentSubscribe** | `initializeRecursively` 遍历时 | Block 从 dead 变为 live | 该 Block 内所有分析被重新调度 |

这三种机制确保：只要有任何一个 Lattice 发生变化，所有依赖它的分析都会被重新调度，直到 fixpoint。

### 9.5 AxisInfoAnalysis 的三个覆盖方法

```cpp
// AxisInfo.cpp:136-179
class AxisInfoAnalysis : public dataflow::SparseForwardDataFlowAnalysis<
                             dataflow::Lattice<AxisInfo>> {
private:
  AxisInfoVisitorList visitors;

  // ─── 覆盖 1: setToEntryState ───
  void setToEntryState(dataflow::Lattice<AxisInfo> *lattice) override {
    propagateIfChanged(
        lattice, lattice->join(
                     AxisInfo::getPessimisticValueState(lattice->getAnchor())));
    // func arg → 读取 tt.contiguity 等参数属性
    // 否则 → contiguity=1, divisibility=1, constancy=1
  }

  // ─── 覆盖 2: visitNonControlFlowArguments ───
  void visitNonControlFlowArguments(
      Operation *op, const RegionSuccessor &successor,
      ArrayRef<dataflow::Lattice<AxisInfo> *> argLattices,
      unsigned firstIndex) override {
    if (auto forOp = dyn_cast<scf::ForOp>(op))
      visitForOpInductionVar(forOp, argLattices);
    else if (auto ws = dyn_cast<gpu::WarpSpecializePartitionsOp>(op))
      visitWarpSpecializeExplicitCaptures(ws, successor, argLattices);
    else
      setAllToEntryStates(...);
  }

public:
  // ─── 覆盖 3: visitOperation ───
  LogicalResult visitOperation(
      Operation *op,
      ArrayRef<const dataflow::Lattice<AxisInfo> *> operands,
      ArrayRef<dataflow::Lattice<AxisInfo> *> results) override {
    // ① 通过 Visitor 分发到具体操作计算逻辑
    AxisInfo curr = visitors.apply(op, operands);

    // ② 算不出结果 → 保守退回到 entry state
    if (curr.getRank() == 0) {
      setAllToEntryStates(results);
      return success();
    }

    // ③ 用 hint attribute 覆盖计算结果
    auto newContiguity = curr.getContiguity();
    auto newDivisibility = curr.getDivisibility();
    auto newConstancy = curr.getConstancy();
    AxisInfo::initDimVectorFromHint(
        op->getDiscardableAttr("tt.contiguity"), &newContiguity);
    AxisInfo::initDimVectorFromHint(
        op->getDiscardableAttr("tt.divisibility"), &newDivisibility);
    AxisInfo::initDimVectorFromHint(
        op->getDiscardableAttr("tt.constancy"), &newConstancy);
    curr = AxisInfo(newContiguity, newDivisibility, newConstancy,
                    curr.getConstantValue());

    // ④ join 到所有结果 Lattice，触发 propagateIfChanged
    for (auto *result : results)
      propagateIfChanged(result, result->join(curr));
    return success();
  }
};
```

这三个覆盖方法构成 AxisInfoAnalysis 的全部自定义逻辑。其余工作（IR 遍历、控制流处理、use-def 链追踪、worklist 调度）由 `SparseForwardDataFlowAnalysis` 和 `DataFlowSolver` 框架自动完成。

### 9.6 SCP 与 AxisInfo 常量系统对比

这是一个容易混淆的点——两套常量系统并行存在：

| | SparseConstantPropagation | AxisInfoAnalysis |
|--|--------------------------|-----------------|
| Lattice 类型 | `Lattice<ConstantValue>` | `Lattice<AxisInfo>` |
| 常量字段 | `ConstantValue::constant` (Attribute) | `AxisInfo::constantValue` (int64_t) |
| 传播方式 | 按 SSA use-def 链，OpFoldResult 折叠 | 按 SSA use-def 链，Visitor 的 getConstantValue |
| 被谁读取 | DCA 的 getOperandValues() | AxisInfoAnalysis 的 Visitor（如 MulIOp 检查乘数 =1） |

**两套并行存在的原因**：SCP 是 MLIR 通用分析，用于常量折叠、分支条件求值等通用场景；AxisInfo 的 constantValue 是 Triton 专用属性，随 contiguity/divisibility 一起在 AxisInfo Lattice 中传播。

## 十一、函数参数上的 `tt.divisibility` 注解（JIT 运行时特化）

在 IR dump 中可以看到函数参数带有 `tt.divisibility = 16 : i32`：

```mlir
tt.func public @coalesced_2d_kernel(
  %in_ptr: !tt.ptr<f32> {tt.divisibility = 16 : i32},     <- 16 字节对齐
  %out_ptr: !tt.ptr<f32> {tt.divisibility = 16 : i32},    <- 16 字节对齐
  %ynumel: i32 {tt.divisibility = 16 : i32},               <- 16 的倍数
  %xstride_in: i32 {tt.divisibility = 16 : i32},           <- 16 的倍数
  %xstride_out: i32 {tt.divisibility = 16 : i32},          <- 16 的倍数
)
```

**这个 `tt.divisibility = 16` 和 AxisInfo 内部的 divisibility（2^62、256 等）不是同一个东西。** 它是 Triton JIT 编译器在 Python 层面对**运行时参数的实际值**做的对齐检测。

### 11.1 工作原理

Triton 的 JIT 编译器在编译 kernel 时，会检查每个参数的实际值（整数或 tensor 指针地址）：

```python
# triton/python/triton/backends/compiler.py
@staticmethod
def parse_attr(desc):
    ret = []
    if "D" in desc:
        ret += [["tt.divisibility", 16]]   # <- 硬编码为 16
    return ret

@staticmethod
def get_int_specialization(arg, **kwargs):
    if arg % 16 == 0 and kwargs.get("align", False):
        return "D"                          # <- 整数是 16 的倍数
    return ""

@staticmethod
def get_tensor_specialization(arg, **kwargs):
    if arg.data_ptr() % 16 == 0 and kwargs.get("align", False):
        return "D"                          # <- tensor 指针是 16 字节对齐
    return ""
```

**特化流程**：

```
Python 参数值检查
    │
    ├── 整数参数: arg % 16 == 0 ? -> 标记 "D"
    ├── tensor 参数: data_ptr % 16 == 0 ? -> 标记 "D"
    │
    ▼
parse_attr("D") -> [["tt.divisibility", 16]]
    │
    ▼
fn.set_arg_attr(cursor, "tt.divisibility", 16)
    │
    ▼
MLIR IR: %arg: type {tt.divisibility = 16 : i32}
```

### 11.2 与 AxisInfo divisibility 的关系

这是**完全不同的两个层次**：

| 层次 | 名称 | 值来源 | 含义 | 举例 |
|------|------|--------|------|------|
| **Python 编译期** | `tt.divisibility`（函数 arg attr） | 运行时参数实际值检测 | "这个参数的**运行时值**是 16 对齐的" | `in_ptr` 地址末尾 4 位为 0 -> `tt.divisibility = 16` |
| **C++ 数据流分析** | `AxisInfo::divisibility` | `highestPowOf2Divisor` 沿 SSA 传播 | "这个 SSA 值的**首元素对齐能力**是 N" | `arange(0, 128)` -> 2^62；`pid * 128 + arange` -> 最终 256 |

**两者如何衔接**：

```
Python 层: %in_ptr 指针是 16 字节对齐
               | (tt.divisibility = 16 作为 arg attr 注入 IR)
C++ AxisInfo 初始化: AxisInfo::initPessimisticStateFromFunc()
               读取 arg attr "tt.divisibility"
               -> divisibility = [16]
               | (作为数据流的悲观初始状态)
               | (后续运算传播并细化: xstride, +arange...)
               -> 最终计算出 ptr tensor 实际的 divisibility
               -> 参与 getNumElementsPerThread 的向量化决策
```

以 `xstride_in` 为例：

```
xstride_in = 2000（运行时值）
  Python 检测: 2000 % 16 = 0 -> 标记 "D" -> tt.divisibility = 16
  AxisInfo 初始化: divisibility = [16]  （保守起点：至少 16 对齐）
  后续运算 (xstride_in * indices_x): 
    AxisInfo 传播: divisibility = gcd(divisibility_xstride=16, divisibility_indices)
    具体值取决于 indices_x 的 divisibility
```

这里的 `16` 是一个**保守的编译期承诺**："这个参数的实际值至少是 16 对齐的，编译器可以基于这个假设做优化"。它作为数据流分析的初始状态，经过 AxisInfo 传播后，最终参与决定 `sizePerThread` 的上限。
